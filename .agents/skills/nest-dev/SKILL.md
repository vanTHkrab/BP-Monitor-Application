## Responsibility

Produces production-ready NestJS code (resolvers, services, types, modules, guards, pipes, Prisma changes) inside `server/app/api-gateway/` that compiles, regenerates `schema.gql` cleanly, and passes type-check and any tests it touched.

You do **not** touch files outside `server/app/api-gateway/` (cross-cutting work must be flagged back with a 2–3 option proposal, not silently executed); hand-create NestJS scaffolding (use `pnpm exec nest g …` so DI wiring and spec files match conventions); hand-edit `src/schema.gql` (it is regenerated by `autoSchemaFile`); hand-edit `package.json` (use `pnpm add` / `pnpm remove` from `api-gateway/` so the lockfile stays consistent); mix Node.js and Python dep bumps in the same change; keep ghost packages (every added dep must include the import that justifies it in the same diff); read `process.env.JWT_SECRET` directly (go through `getJwtSecret()`); validate inputs manually inside resolvers/services (class-validator decorators on `@InputType` fields are mandatory, including `@IsEnum(...)` on enum fields, because the global `ValidationPipe` uses `whitelist` + `forbidNonWhitelisted` and any undecorated field 400s before reaching the resolver); return error objects from resolvers (throw `HttpException` subclasses so the `errorFormatter` stamps `extensions.code` for the mobile + web clients); make one-sided changes to the Redis AI wire contract (`analyze_bp_image` / `analyze_bp_image.reply`) — refuse and flag back, both sides must change in the same task; edit the database by hand or skip `pnpm prisma migrate dev` for schema changes; run the canonical test suite as the PR gate, write commit messages, or open PRs (downstream agents own those); drive-by refactor unrelated modules (one feature module per change); or silently pick one approach for non-trivial work (schema change, new auth/session flow, cross-module work, perf work with multiple possible fix layers, refactors >5 files) — present 2–3 options with pros / cons / when-each-fits and wait for the user to choose.

Pre-condition: the dispatcher has confirmed the task is scoped to the api-gateway. If the brief itself names `client/`, `web/`, or `ai-service/` paths, halt at Step 1.

## Step 1 — Shape the task and surface trade-offs

Confirm scope, locate the target module, and decide whether the task is mechanical or non-trivial.

```text
1. Read the brief carefully. If it touches client/, web/, or ai-service/, emit BLOCKED
   (out-of-scope refusal) — do not write any code.
2. Locate the target feature module under server/app/api-gateway/src/. Read its
   *.module.ts, *.resolver.ts, *.service.ts, *.types.ts so new code matches local
   convention.
3. Classify the task:
   - Mechanical (rename, off-by-one, single-file fix the brief already specifies)
     → proceed to Step 2.
   - Non-trivial (schema change, new auth/session flow, cross-module work, perf
     work with multiple possible fix layers, refactor >5 files)
     → produce 2–3 options with pros / cons / when each fits, emit BLOCKED
       (unresolved trade-off), wait for user choice. No code written.
4. For new modules/resolvers/services/guards/pipes/interceptors, plan to scaffold
   via `pnpm exec nest g …` rather than hand-creating.
5. For Prisma schema changes, plan the migration via `pnpm prisma migrate dev` —
   never edit the DB by hand.
6. If src/ai/ is in scope, confirm the matching ai-service change is part of the
   same task. If not, emit BLOCKED (wire-contract one-sided).
```

## Step 2 — Scaffold via the Nest CLI

Generate new artifacts through the CLI so DI wiring, spec files, and structure match the rest of the project. Reorganising files inside an existing module is a manual refactor — the CLI generates, it does not move.

```bash
# from server/app/api-gateway/
pnpm exec nest g module foo
pnpm exec nest g service foo
pnpm exec nest g resolver foo
# same CLI for guards, pipes, interceptors

# dependency changes (never hand-edit package.json)
pnpm add <pkg>
pnpm add -D <pkg>
pnpm remove <pkg>
```

After install, verify `pnpm-lock.yaml` changed and commit it in the same diff as the manifest. Every added dep must have its justifying import in the same diff.

## Step 3 — Implement under project conventions

Apply the api-gateway's conventions exactly. Each bullet below is load-bearing — skipping any one of them silently breaks a downstream consumer.

```text
Module shape:
- xxx.module.ts + xxx.resolver.ts + xxx.service.ts + xxx.types.ts.
- Resolver injects the service; service injects PrismaService via DI.

Types & validation:
- Update *.types.ts first with @ObjectType / @InputType.
- EVERY @InputType field needs a class-validator decorator. @Field(() => MyEnum)
  alone is not enough — enums require @IsEnum(MyEnum).
- Never validate manually inside resolvers/services; the global ValidationPipe
  (whitelist + forbidNonWhitelisted + transform) handles it.

Auth:
- @UseGuards(GqlAuthGuard) + @CurrentUser() on anything authenticated.
- Read JWT secret only through getJwtSecret() in src/auth/auth.config.ts.
- Respect the throttled lastActiveAt write pattern (≥5 min) — don't add per-request
  DB writes elsewhere either.

Errors:
- Throw HttpException subclasses: UnauthorizedException, ForbiddenException,
  ConflictException, BadRequestException, NotFoundException.
- The errorFormatter in app.module.ts maps HTTP status → extensions.code, which
  the mobile + web clients branch on. Never return error objects from resolvers.
- User-facing HttpException messages stay Thai (root rule 7 exception). Code,
  comments, logs, and commit-style strings stay English.

Persistence:
- Always via DI-injected PrismaService.
- Composite atomic writes use prisma.$transaction([...]).
- Prefer findUnique over findFirst on indexed columns.
- Image quality score → prisma.image.updateMany (not update) so a swept row
  doesn't fail analysis.

Redis:
- Optional at boot, lazy-connects with errors suppressed.
- Consumers check redis.status === 'ready' and degrade if not.
- Login throttler falls back to per-process in-memory map — keep that.

AI wire contract:
- If touching src/ai/, the matching change in
  server/app/ai-service/src/ai_service/handlers.py must be planned in the same
  task. Payload shape lives in src/ai/ai.process.ts.
- One-sided change → refuse and emit BLOCKED.

Performance:
- Avoid N+1 by batching with Promise.all or DataLoader.
- Don't add per-request DB writes outside the existing throttled patterns.

Storage:
- S3 writes go through src/storage/. StorageCleanupService is a daily @Cron
  orphan-image sweep — don't break that contract.

Dependencies:
- No new top-level deps without a same-diff import.
- Remove deps you stop importing in the same change.
- Never mix Node.js + Python dep bumps.
```

## Step 4 — Verify

Type-check, regenerate the schema, and run only the tests covering files this task touched. The canonical test suite is `tester`'s job — not this agent's.

```bash
pnpm --dir server/app/api-gateway exec tsc --noEmit
pnpm --dir server/app/api-gateway start:dev     # brief — confirm schema.gql regenerates cleanly
pnpm --dir server/app/api-gateway test          # only suites covering touched files
pnpm --dir server/app/api-gateway prisma migrate dev   # if schema.prisma changed
```

For new behavior in `src/auth/`, add at least one unit test on the service path (the module has no tests yet). Record verification results in the verdict body.

## Step 5 — Emit the verdict

### On success — DONE

```text
## nest-dev: DONE

Task: <one-line restatement>
Files changed:
- <path> — <what changed>
Schema impact: <none | regenerated; diff under server/app/api-gateway/src/schema.gql>
Verify:
- tsc --noEmit: <PASS | FAIL output>
- prisma migrate dev: <ran | not needed>
- tests touched: <list, with PASS/FAIL>
Trade-offs taken: <one line — what was chosen and what was given up>
Hand off to: tester
```

### On unresolved trade-off — BLOCKED

```text
## nest-dev: BLOCKED

Reason: non-trivial task — proposing 2–3 options before coding.
Options:
1. <name> — <pros> / <cons> / <when this fits>
2. <name> — <pros> / <cons> / <when this fits>
3. <name> — <pros> / <cons> / <when this fits>
Recommendation: <option N, one line why>

Waiting for user choice. No code written.
```

### On out-of-scope refusal — BLOCKED

```text
## nest-dev: BLOCKED

Reason: <cross-cutting | wire-contract one-sided | out of api-gateway/ scope>
Boundary: <which rule from CLAUDE.md was about to be crossed>
Next step: <what the user / dispatcher should do — e.g. open a paired task in ai-service, or split the change>

No files modified.
```

Hand off to `tester` on `DONE`. The full downstream chain is `tester` → `pr-write` → `pr-review` → `gh-stack`, routed by the dispatcher.
